Pixel Image Serving And Google's ModPageSpeed - Quick Checklist

Posted at Sat, 30 Jul 2016 23:38:43 GMT



This is all about PIXEL-IMAGES,


While pixel-images themselves are all about an out-going request,
with little to none caring of the actual response -

- Due to compatibility issues, browser pre-rendering enhancements,
mimetype-sniffing, client-side caching, cross-domain limitations,
and content/visibility base loading such as lazy-loading and content-deferring -

- In most cases we actually need to, also, (ridiculously) ensure that the pixel-image has rendered perfectly.





Note:
Pixel-image is an archaic way of notifying the server-side of *stuff*.
If possible, try avoid the pixel-image in favour of navigator.sendBeacon, which is superior in every-way (for supported browsers). If you must- then at-least have the pixel-image as fallback to the sendBeacon method




PageSpeed "no defer" attribute is a FLAG!

.

Including data-pagespeed-no-defer or pagespeed_no_defer (for older versions of ModPageSpeed)
will exclude the image from the PageSpeed resource-defer engine,

It has no use in the HTML itself, and is only a "flag" used for the PageSpeed-extension running at server-side,
the attribute it will be removed from the HTML onced the source has finished processed, so if you want to make sure you've placed the attribute, load your page using ..../my_page.php?ModPagespeed=off attribute (and view the page's source).

Some web-hosting plans such as GoDaddy's economy-plan, includes an old-version of ModPageSpeed,
which does not recognise the data-pagespeed-no-defer attribute,

since pagespeed_no_defer won't be removed from the ModPageSpeed source anytime soon,
you'll be better use that instead -- it will be removed from HTML source anyway..

HTML5, XHTML Compatibility, and client side rendering best-practices for image-pixel:

<span style="visibility:hidden; z-index:-9999;"><img data-pagespeed-no-defer="true" pagespeed_no_defer="true" width="1" height="1" border="0" alt="" rel="nofollow,noindex" src="___PIXEL_URL___"/></span>




And How About Some Pixel-Serving Checklist...




  1. Minimal (BUT True!) image-content:


    • BASE64 decode this little thing R0lGODlhAQABAAAAACH5BAEKAAEALAAAAAABAAEAAAICTAEAOw==
      and serve it as is.


    • use correct response-mimetype: image/gif



  2. Don't use /assets/ folder for your pixel-image
    IIS, NodeJS, Jetty and other servers might provide you a bootstrap folder structure with a /static/, /resources/ or /assets/ folder,
    Placing your image under there might requires you of resolving all sort an caching issues:


    • make sure to send the image with HTTP Status 200 and not HTTP Status 304


    • remove ETAG



  3. Don't use a real GIF file
    Reading an image file from disk is pricey!
    decode the BASE64 above, And keep it in a variable,

    If possible, place the content (either binary or BASE64-encoded) at a persistence state,
    for the server-lifetime.

    RAM APIs that might also work for you: APC, APCu, MemCache, Memcached or Redis

  4. No cache headers:


    • Expires: Sat, 26 Jul 1997 05:00:00 GMT


    • Cache-Control: no-store,no-cache,must-revalidate,pre-check=0,post-check=0


    • Pragma: no-cache



  5. A browser-friendly binary content, explicitly:

    • Include multi-part header for faster handling
      Accept-Ranges: bytes

    • prevent mimetype sniffing (first few bytes of the file) and just download the file.
      X-Content-Type-Options: nosniff

    • Include multi-part header for faster handling
      Content-Transfer-Encoding: binary

    • Include multi-part header for faster handling
      Content-Disposition: inline;filename="pixel.gif"



  6. CORS And Tainting Issues
    If you'll serve a pixel-image to a different (sub-)domain,
    Explicit enable the (sub-)domains using the CORS headers.
    Since pixel-image are all about the initial-request, you might even use the * syntax:
    Access-Control-Allow-Origin: *, Access-Control-Allow-Credentials: true and Timing-Allow-Origin: *.

    *It will also ensure you serve a resource that will not cause canvas-tainting.
    *Read more at https://developer.mozilla.org/en-US/docs/Web/HTML/CORS .
